Conventions de calcul partagées : (1) « hors annulées » = filter status <> 'cancelled' partout sur subscriptions. (2) Investisseurs total = count investors where deleted_at is null. (3) Les métriques de période utilisent signed_at (collecte/subs/investisseurs) ou sah_created_at (inscrits), bornées par resolvePeriod (src/lib/period.ts), période par défaut = 1 mois glissant, comparées à la fenêtre précédente équivalente ; delta() calcule deltaAbs et deltaPct (null si base précédente nulle).

Divergences / points à clarifier (3) : 
1. STATUT PROJET non fiable — projR.open filtre status in ('open','funding') mais la mémoire projet indique que tous les projets sont synchronisés en 'open' (mapping statut à faire). → 'Projets ouverts' (KPI) et 'X en collecte' (accès rapide) en QUARANTAINE.
2. ATTRIBUTION BREACH non validée — détection via bonus_code ilike '%breach%' OR breach_level is not null ; la mémoire signale le sous-parrainage non traçable (bonus_codes sans propriétaire, 0 sous-filleul). Total direct plausible mais attribution incertaine. → 'Leads BREACH' (KPI), 'X € collectés' BREACH, et 'X leads' (accès rapide) en QUARANTAINE.
3. DOUBLONS — l'en-tête (X investisseurs / X € collectés) duplique deux KPIs ; les hints d'accès rapides (Projets, BREACH) dupliquent des KPIs. Proposé DÉPLACER pour l'en-tête (cosmétique) et QUARANTAINE pour les hints concernés (héritent du défaut de fiabilité de leur source).

Mappings booléens confirmés (mémoire) : 'Profil complété' = registration_complete (infos perso renseignées), 'Onboardés (KYC)' = onboarding_complete — les 2 seuls booléens reçus de SAH, cohérents avec le funnel.

Limite mineure non bloquante : le graphique byMonth fait limit 12 sur les mois PRÉSENTS dans subscriptions (mois sans collecte absents → trous possibles dans la courbe), et le stepRate du funnel 'Ont investi' croise un numérateur (subscriptions) et un dénominateur (investors.onboarded) de tables différentes (peut dépasser 100% si un investisseur non-onboarded a souscrit).

(2) INSCRITS / COMPLETS / INVESTISSEURS / COLLECTE / ticket / conversion viennent de la BASE SAH (tables investors + subscriptions), attribués à une régie par bonus_code : 'SEVEN-BREACH%' → Meta, '%VIP%' → Google (AD_CODE_PATTERNS, ads-acquisition.ts) ; les codes partenaires/CGP sont exclus.

POINT CRITIQUE de fiabilité : le code lui-même considère les 'résultats'/conversions du pixel comme GONFLÉS et les ignore volontairement dans le bloc 'coût réel' (blended.ts, commentaire 'On ignore les conversions gonflées du pixel'). Conséquence : toutes les métriques dérivées des 'résultats' régie (coût/résultat global, trends résultats, top/flop par CPA, alerte CPA outlier, répartition par plateforme coût/résultat) sont structurellement moins fiables que le bloc 'Coût réel croisé SAH' qui, lui, divise la dépense par les vrais inscrits SAH. D'où les nombreux DÉPLACER sur les métriques basées 'résultats pixel'.

BIAIS TEMPOREL (signalé dans l'UI sous CostTable) : investisseurs/collecte sont comptés par signed_at dans la fenêtre, alors que inscrits/complets sont par sah_created_at — sur une période récente le funnel est partiel (le closing prend des semaines). Les taux 'Conv. inv.' et la 'Rentabilité' sur fenêtre courte sont donc à lire avec prudence (numérateur et dénominateur ne portent pas sur la même cohorte de personnes).
